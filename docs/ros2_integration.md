# ROS 2 Integration

clankeRS talks to ROS 2 through the `clankers-ros2` crate, which has **two
backends behind one API** selected at compile time:

| Backend | Feature | Needs ROS install? | Used by |
|---------|---------|--------------------|---------|
| Simulated in-memory bus | `sim` (default) | No | CI, `cargo build`, all examples/tests |
| Real rclrs / DDS | `ros2` | **Yes** (+ colcon) | Live robots, `ros2 topic` interop |

`RobotNode`, `Publisher`, `Subscriber` are identical across both — only the
transport changes.

```rust
use clankers::prelude::*;

let node = RobotNode::new("my_node").await?;
let mut sub = node.subscribe::<ImageMsg>("/camera/image_raw", QosProfile::sensor_data()).await?;
let det = node.publish::<DetectionArray>("/detections", QosProfile::default()).await?;
```

## Default: simulation backend

Nothing to install. This is what `cargo build`, CI, and the
[`camera_replay`](../crates/clankers/examples/camera_replay.rs) golden path use.
Pub/sub is an in-memory broadcast bus; messages are serialized as JSON.

## Real ROS 2 (rclrs) — status

> **Status: implemented but not yet verified on hardware.** The rclrs backend
> ([`rclrs_backend.rs`](../crates/clankers-ros2/src/rclrs_backend.rs)) and the
> `sensor_msgs/Image` bridge ([`bridge/image.rs`](../crates/clankers-ros2/src/bridge/image.rs))
> are written against **rclrs 0.7** but can only be compiled inside a ROS 2 +
> colcon environment, which the maintainer has not yet run end-to-end. Treat the
> commands below as the intended flow, to be validated on a ROS box.

### Why it can't be a plain `cargo` dependency

`rclrs` links the ROS 2 C libraries, and the message crates (`sensor_msgs`,
`std_msgs`, `builtin_interfaces`) are **generated by colcon** — they are not on
crates.io. So they are declared in [`package.xml`](../crates/clankers-ros2/package.xml)
(read by `colcon-ros-cargo`), **not** in `Cargo.toml`. This is why the default
`cargo build` stays completely ROS-free.

### Build steps (Ubuntu 22.04 + ROS 2 Humble)

```bash
# 1. Install ROS 2 Humble (ros-humble-ros-base) — or use .devcontainer/
sudo apt install ros-humble-ros-base python3-vcstool libclang-dev

# 2. Bootstrap the ros2-rust workspace (rclrs + message crates)
source /opt/ros/humble/setup.bash
bash scripts/setup_ros2_ws.sh

# 3. Build clankeRS with the real backend
source ros2_ws/install/setup.bash
cargo build -p clankers-ros2 --features ros2
```

### Smoke test

```bash
# Terminal A — publish real sensor_msgs to /camera/image_raw
cargo run -p ros2_pubsub_minimal --features ros2

# Terminal B — confirm it on the ROS graph
source /opt/ros/humble/setup.bash
ros2 topic list                 # should include /camera/image_raw
ros2 topic echo /camera/image_raw --no-arr
```

### Transport note (MVP)

The current rclrs backend carries every clankeRS message as `std_msgs/String`
holding the message JSON, which makes it generic over any message type. Typed
`sensor_msgs/Image` publishing (via the bridge in
[`bridge/image.rs`](../crates/clankers-ros2/src/bridge/image.rs)) is the next
step for full wire-compatibility with stock ROS nodes; `DetectionArray` needs a
custom `.msg` definition (tracked in the roadmap).

## QoS profiles

```rust
QosProfile::sensor_data()  // best-effort, shallow depth — cameras/lidar
QosProfile::default()      // reliable, volatile — commands/state
```

Under the `ros2` backend these map to rclrs reliability/durability/depth; under
`sim` they are currently advisory.
